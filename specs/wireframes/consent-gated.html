COPPA Consent Gating

Screen: consent-gated | FR: FR-SAFE-004 | Persona: PER-001 (Student under 13) | Journeys: JRN-003, JRN-005

Layout Zones

1
Access Blocked Banner (full-screen gate — student view) — Shown instead of the normal chat interface when consent is pending or denied. Message: "Your parent or guardian needs to approve your access to [Platform Name]." Student-facing language is simple and non-alarming. Shows: the parent's email (partially obscured: "An email was sent to j***@example.com"), estimated wait time ("Your parent received an email. Once they approve, you'll be able to start."), and a "Need help?" link.
2
Consent Status Indicator — Visual status tracker (not interactive): Step 1: "Email sent to parent" (completed). Step 2: "Parent verifies identity" (pending). Step 3: "You're all set!" (locked). Updates in real time via WebSocket push when parent completes consent.
3
Parent Consent Flow (parent's browser — separate URL) — Parents receive an email with a link. Clicking opens a page (no login required at start) with: plain-language explanation of what data is collected, how it's used, who can see it. Identity verification step (text-plus or knowledge-based auth). "Grant Access" and "Deny Access" buttons. Legal footer: links to Privacy Policy and COPPA rights notice.
4
Supplementary App Consent (conditional) — When a teacher enables a new app for a K-5 classroom, a separate consent notice appears for just that app. Uses the same parent flow but scoped to the specific app: "Ms. Torres has enabled [App Name] for your child's class. This app [description]. Data collected: [app-specific list]."

Trigger Conditions

TRIGGER Student added to K-5 classroom: Teacher adds students to a classroom with grade band K-2 or 3-5. The system detects that the grade band includes students who may be under 13. Per AC-SAFE-004-01: COPPA consent workflow is initiated automatically. Parents receive email within 5 minutes of student being added.

TRIGGER Student attempts to log in without consent: SSO callback returns 403 with { "error": "consent_required", "consent_id": "uuid" }. The student is redirected to this gating screen instead of the chat interface. Per ssoCallback API error codes.

TRIGGER New app enabled (supplementary consent): Teacher enables a new app that processes student data. Per AC-SAFE-004-02: supplementary consent notification sent to parents. Students in that classroom who are under 13 see the app as "locked" until parent approves for that specific app.

TRIGGER Consent revoked: Parent revokes consent (FSM-004 → revoked). Student's active session receives a WebSocket system_notification: "Your access has been changed. Please speak to your teacher." Student is redirected to this gating screen.

API Endpoints

API Check consent status (student): GET /api/v1/consent/{consent_id}/status — Returns { "status": "pending" | "granted" | "denied" | "revoked", "parent_email_masked": "j***@..." }. Called on page load to render the correct state.

API Parent consent submission: POST /api/v1/consent/verify — Body: { "consent_id": "uuid", "decision": "granted" | "denied", "verification_token": "..." }. Per FSM-004 VERIFICATION_SUCCESS / CONSENT_DENIED events. If granted: student access is unlocked. If denied: student is locked out.

API WebSocket consent granted push (student): When parent grants consent, server pushes system_notification with type consent_granted to the student's waiting screen. Student is automatically redirected to the chat interface — no page refresh needed.

API Resend parent notification: POST /api/v1/consent/{consent_id}/resend — Allows teacher or admin to resend the parent email. Rate-limited: max once per 24 hours. Shown to teacher in Mission Control for students awaiting consent.

State Transitions (FSM-004: COPPA Consent Flow)

FSM Initial: pending → INITIATE_CONSENT event → notified. send_notification action emails parent.

FSM Parent responds: PARENT_RESPONDS event → verified. Parent is on the verification page.

FSM Verification outcomes:

  • VERIFICATION_SUCCESS → granted. grant_consent action fires. Student access unlocked. WebSocket push to student.
  • VERIFICATION_FAILED → back to notified. Parent can retry (max 3 attempts).
  • CONSENT_DENIED → denied. lock_student action fires. Student sees denied state on gating screen.

FSM Timeout: 7 days with no response → denied (treated as denial). REMINDER_SENT at 48h (stays in notified).

FSM Re-entry: From denied or revoked: RESUBMIT event → back to pending. The consent cycle can repeat.

User Exit Paths

  • Consent granted (parent action): Student's gating screen auto-redirects to chat interface via WebSocket push. No student action needed.
  • Log out button (student): Student can log out of the gating screen. Consent remains pending. Next login shows the same gating screen.
  • "Need help?" link: Opens a static help page explaining the consent process. Links to district contact info. Opens in new tab.
  • Parent: Grant Access (consent flow): POST /consent/verify with decision=granted. Parent sees confirmation: "Your child can now access the platform." Parent's browser stays on confirmation page.
  • Parent: Deny Access (consent flow): POST /consent/verify with decision=denied. Parent sees: "Access has been denied. Your child will see a message to speak with their teacher. You can change this at any time." Parent's browser stays on confirmation page.

Error Recovery

ERROR SSO callback 403 (unexpected): If the gating screen itself fails to load (GET /consent fails), show a generic: "We're having trouble loading your account. Please ask your teacher for help." Do not expose consent_id to student.

ERROR WebSocket not available (student waiting screen): Student must manually refresh the page to check if consent has been granted. Show: "Your page will update automatically when your parent approves. You can also refresh to check."

ERROR Parent verification fails 3 times: Form shows "Too many failed verification attempts. Contact your district administrator for assistance." Consent remains in notified state. Admin can manually trigger a new verification link.

ERROR Parent link expired (7-day timeout → denied): Parent clicks old link and sees: "This consent request has expired. Please contact your child's teacher to restart the process." Resend is not available to parent directly — only teacher/admin can trigger resend.

Accessibility Notes

  • Student gating screen uses simple, grade-appropriate language. K-2 reading level for K-5 students. Avoid legal jargon.
  • Status tracker (steps 1-3) uses role="list" with each step as a role="listitem". Completed steps have aria-label="Step 1: Email sent to parent — completed".
  • Parent consent form: all data collection items are disclosed in a scannable list (not a wall of text). Checkboxes for explicit consent are optional (consent is a single decision, not per-item).
  • Parent "Grant Access" button uses aria-label="Grant [Platform Name] access for my child" to be unambiguous.
  • Parent "Deny Access" button is secondary priority (smaller, but fully accessible). Screen reader reads both buttons before the user commits.
  • WebSocket-triggered redirect announces to screen reader via aria-live="assertive": "Your parent has approved access. Redirecting to your classroom now."
