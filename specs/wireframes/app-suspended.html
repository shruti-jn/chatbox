App Suspension Notification

Screen: app-suspended | FR: FR-APP-004 | Personas: PER-001 (Student), PER-002 (Teacher) | Journeys: JRN-001, JRN-005

Layout Zones

1
Suspended App Card (in chat thread — Student view) — Replaces the active app card in-place. Gray background, lock icon. Message: "This app has been temporarily disabled by your district." No error styling (not the student's fault). App name shown. No retry option. Chat conversation is unaffected and continues normally below/around the card.
2
AI Acknowledgment Message (in chat thread — Student view) — The AI sends a message in the conversation acknowledging the app is unavailable: "I see [App Name] is no longer available right now. We can continue our conversation — what would you like to do instead?" Keeps the learning session alive.
3
Mission Control Warning Banner (Teacher view — teacher-dashboard) — A yellow banner or toast at the top of the teacher's Mission Control: "[App Name] has been disabled by your district. Active sessions have been terminated." Includes timestamp and a "View affected students" button that filters the student grid to show students who had the app active.
4
Admin Suspension Panel (District Admin view — admin-catalog) — After suspending, the app's catalog entry shows "Suspended" status badge, suspension timestamp, and a "Reinstate" button. Reason field (from the POST body) is shown for audit purposes.

Trigger Conditions

TRIGGER District admin suspends app: Admin calls POST /api/v1/admin/district/{id}/apps/{app_id}/suspend (or the platform admin calls POST /api/v1/admin/apps/{app_id}/suspend for a platform-wide suspension). Per AC-APP-004-01.

TRIGGER App tool endpoint unresponsive (30s): The system's health check detects app tool endpoint failure. App status auto-transitions to "degraded". Per AC-APP-004-02. The AI stops invoking the app's tools. A degraded (not suspended) warning appears for teachers.

TRIGGER Real-time push (no polling): The WebSocket push to affected clients is immediate. Students with the app active see the card change state in real time without any page action. Teachers receive the Mission Control warning via WebSocket push simultaneously.

API Endpoints

API Suspend app (admin): POST /api/v1/admin/district/{district_id}/apps/{app_id}/suspend — Body: { "reason": "security review" }. Side effects: updates catalog status, terminates all active instances, notifies teachers via WebSocket, students see disabled message. Returns immediately (side effects are async).

API Platform-wide suspension: POST /api/v1/admin/apps/{app_id}/suspend — Used by platform admin. Same behavior but across all districts.

API WebSocket (student): WebSocket /ws/chat/{session_id} receives app_event with type app_suspended and instance_id. Frontend replaces the app card UI in-place. AI receives the same event and generates an acknowledgment message.

API WebSocket (teacher): WebSocket /ws/teacher/{classroom_id} receives student_activity event with type app_suspended. Mission Control banner appears.

API Reinstate (admin): POST /api/v1/admin/district/{district_id}/apps/{app_id}/reinstate (implied by FSM-005 REINSTATE event). App returns to live state. Students can invoke the app again in future sessions (existing terminated instances are not restored).

State Transitions

FSM App Instance FSM-002: SUSPEND_REQUESTED event → active instance transitions from active to suspended. Entry action: show_suspended_message pushes the gray disabled card to student's chat. From suspended: UNSUSPEND event → back to active (future sessions only, not current terminated instances).

FSM App Review Pipeline FSM-005: SUSPEND event on a live app → suspended. Entry action: disable_everywhere fires. From suspended: REINSTATE → live or DEPRECATE → deprecated.

FSM Student Conversation FSM-001: The TEACHER_PAUSE or SESSION_END events are not triggered by app suspension. The conversation itself is unaffected and remains in active state. Only the app_embedded sub-state is affected.

User Exit Paths

  • Student — continue conversation: Chat input remains active. Student can type a response to the AI's acknowledgment message. Learning session continues without the app.
  • Student — no action available on card: The suspended app card has no buttons. Student cannot retry, reload, or dismiss it. The card persists in the conversation history as a record of the interruption.
  • Teacher — "View affected students": Filters Mission Control grid to show students who had the app active. Teacher can then message affected students individually.
  • Teacher — dismiss warning banner: Banner closes. It does not re-appear until another app is suspended. The Mission Control grid continues to show which students were affected (their cards show the app as inactive).
  • Admin — Reinstate: Triggers reinstate flow. App returns to live status. Teachers and students are notified via WebSocket.

Error Recovery

ERROR WebSocket down (student) during suspension: Student does not see immediate card update. When WebSocket reconnects, client queries session state and renders suspended card retroactively. The AI's acknowledgment message is delivered on reconnect.

ERROR WebSocket down (teacher) during suspension: Teacher does not receive real-time warning. On next page load or WebSocket reconnect, the suspension event is delivered. Mission Control shows the banner retroactively.

ERROR Reinstate POST fails: Admin sees inline error: "Could not reinstate app. Try again." App remains in suspended state. No student or teacher notifications sent for a failed reinstate.

ERROR App degraded (not suspended): If the health check detects a degraded app (not a full admin suspension), students see a softer warning: "This app is experiencing issues. Try again in a few minutes." Retry button available. AI does not invoke the app's tools until health check passes.

Accessibility Notes

  • Suspended app card uses role="status" and aria-label="[App Name] is temporarily disabled by your district". Screen reader announces the change when the card state updates in-place.
  • The AI's acknowledgment message uses the standard ai_message component (already accessible). No special treatment needed.
  • Mission Control warning banner for teachers uses role="alert" and aria-live="assertive" (this is urgent for teachers who need to respond).
  • Suspended card has no interactive elements. Tab order skips it. The chat input below remains the focused interactive element.
