Crisis Resources

Screen: crisis-resources | FR: FR-SAFE-009 | Persona: PER-001 (Sam the Student) | Journeys: JRN-001

Layout Zones

1
Crisis Message Card (full-width, top of chat thread) — Replaces or pins above the chat input. Warm, non-alarming background (soft blue, not red — avoid panic). Message: "It sounds like you might be going through something really hard. You are not alone." No AI branding on this card — it should feel like a human outreach.
2
Crisis Resource Contacts (center, large tap targets) — Two primary resources, always shown: 1. 988 Suicide & Crisis Lifeline: "Call or text 988 — available 24/7, free and confidential" 2. Crisis Text Line: "Text HOME to 741741" Each row has a phone/text icon, large font, and a tap/click action. On mobile: tap-to-call / tap-to-text. On desktop: copy-to-clipboard with confirmation.
3
School Contact (conditional) — If the school/district has configured a counselor contact, show: "You can also talk to your school counselor: [Counselor Name] — [contact method configured by district]." Falls back to "Talk to a trusted adult at your school." if no counselor configured.
4
Return to Chat (below resources) — A single muted link: "Return to your conversation." Intentionally de-emphasized so the crisis resources are the visual focus. Student must consciously choose to dismiss this screen.

Trigger Conditions

TRIGGER Safety pipeline detection: The content safety classifier detects suicide/self-harm indicators in a student's message. Detection must occur within 200ms. The pipeline blocks the original message from being processed by the LLM and immediately pushes the crisis card to the student via WebSocket.

TRIGGER No student action required: The screen is pushed — not navigated to. The crisis card appears at the top of the student's existing chat thread. It is not a new page; it overlays the chat with zero delay. Per AC-SAFE-009-01.

TRIGGER Concurrent teacher alert: At the same moment the crisis card appears for the student, the teacher receives a priority safety_alert push on their Mission Control WebSocket. The teacher and student views update simultaneously.

API Endpoints

API Crisis card delivery (server push): No client-initiated GET for this screen. The card is pushed via WebSocket /ws/chat/{session_id} with event type crisis_intervention. Payload includes: resource links (pre-configured), school counselor info (if set), audit event ID.

API Tap-to-call / tap-to-text (mobile): Standard tel:988 and sms:741741?body=HOME URI schemes. No API call. Native OS handles.

API Log crisis resource interaction: POST /api/v1/safety-alerts/{alert_id}/events — Logs when student taps a resource link (e.g., { "event": "resource_clicked", "resource": "988_lifeline" }). Used in audit trail. Fire-and-forget; does not block UI.

API Teacher alert: Server-side: WebSocket /ws/teacher/{classroom_id} push with event type safety_alert and severity: "critical". Teacher's Mission Control displays priority alert simultaneously with the student seeing the crisis card.

State Transitions

FSM Student FSM-001: The student's conversation was in active state when the flagged message was sent. The crisis intervention does not directly change the FSM state — the conversation technically remains in active state but the AI response pipeline is blocked for this message.

FSM Teacher FSM-006: Safety pipeline triggers SAFETY_ALERT event → teacher transitions from monitoring to alert_triggered. send_email_notification action fires for critical severity. Teacher must click REVIEW_ALERT to transition to reviewing.

FSM On "Return to chat": Crisis card dismisses. Student's conversation resumes. The next message the student sends goes through the full safety pipeline again — the intervention does not grant any bypass.

User Exit Paths

  • Tap 988 (mobile): Launches phone call. Student leaves the app. App remains in background.
  • Tap Crisis Text Line (mobile): Opens SMS app pre-filled with "HOME" to 741741. App remains in background.
  • Copy number (desktop): Copies to clipboard. Toast: "Copied to clipboard." Student remains on screen.
  • "Return to your conversation" link: Dismisses crisis card. Chat input becomes active again. The flagged message is NOT shown in the transcript — it was blocked. The chat resumes from the last AI response before the flagged message.
  • Close browser/tab: No API call. Session timeout (30 min inactivity) eventually fires SESSION_END.

Important: There is no "X" close button on the crisis card. The student must use the "Return to your conversation" link to make the deliberate choice to exit. This is intentional friction per California SB 243.

Error Recovery

ERROR Safety pipeline unavailable: Per FR-MON-002 External Dependencies: if the classifier is unavailable, messages are flagged as "unclassified" and teacher visibility is increased. In this degraded state, the crisis card is NOT shown automatically — teacher must review manually. The student sees the AI respond normally (potentially to a crisis message). This is a known acceptable failure mode, logged as a critical system event.

ERROR WebSocket down (student side): Crisis card cannot be pushed. Server detects WebSocket failure and falls back to inserting the crisis card as a message in the conversation history so it appears on next load or WebSocket reconnect.

ERROR Crisis log POST fails: Silently retried in background. Never blocks the UI. The crisis card remains displayed regardless of log success.

Accessibility Notes

  • Crisis card uses role="alert" and aria-live="assertive" so it is announced immediately by screen readers when it appears: "This is important. [Message text]. Resources available: 988 Suicide & Crisis Lifeline, Crisis Text Line."
  • Resource buttons have explicit aria-label: "Call or text 988 — Suicide and Crisis Lifeline" and "Text HOME to 741741 — Crisis Text Line".
  • Color contrast for all text meets WCAG AA 4.5:1 minimum. Background is NOT red (which can be alarming and inaccessible for color-blind users).
  • Font size for resource contacts is minimum 18px (large enough to be legible under stress).
  • "Return to conversation" link is deprioritized visually but is fully keyboard accessible and announced after the resource buttons in tab order.
  • No auto-dismiss timer. Screen persists until student actively exits.
