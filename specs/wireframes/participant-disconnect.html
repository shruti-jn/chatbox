Participant Disconnect

Screen: participant-disconnect | FR: FR-COLLAB-004 | Personas: PER-001 (Student) | Journeys: JRN-002

Layout Zones

1
Disconnect Notification (in collaborative session view — remaining participants) — Inline notification below the participant roster: "[Student Name] disconnected. Waiting for them to reconnect (up to 5 minutes)." Amber styling. A 5-minute countdown timer shows time remaining. If turn-based: "It's [Other Student]'s turn. The game is paused while [Disconnected Student] reconnects." If concurrent: "You can keep going — [Disconnected Student] will catch up when they reconnect."
2
Participant Roster (in session header) — List of all participants with connection status dots: green = connected, amber = reconnecting, gray = disconnected. Disconnected participant's avatar grays out with a clock icon. Roster is always visible in the collaborative session view.
3
Reconnection View (disconnected student's browser) — When a student's connection drops, their own view shows a reconnection overlay: "You got disconnected. Reconnecting..." with a spinner. Auto-reconnects up to 5 attempts at 5-second intervals. On success: overlay dismisses, session state is delivered, game resumes. On failure (5 min timeout): overlay shows "You couldn't reconnect in time. The session may have continued without you." with a "Return to chat" button.
4
All-Disconnected Threshold (edge case) — If all participants disconnect simultaneously (e.g., class ends mid-session), the session transitions to ending state via the ALL_DISCONNECTED event. When any participant returns, they see: "Your session ended because everyone disconnected. Here's a summary of what you accomplished." Summary card (same as collab-session-end).

Trigger Conditions

TRIGGER WebSocket connection drops: The server detects a participant's WebSocket disconnection (no ping response). PARTICIPANT_DISCONNECTED event fires in FSM-003. Per AC-COLLAB-004-01: remaining participants see "[Student name] disconnected" and the session continues.

TRIGGER Browser closed / tab closed / network loss: Any of these causes the WebSocket to drop. The server distinguishes a graceful close (CLOSE frame) from a network loss (heartbeat timeout after ~15s). Both are treated as disconnections for session purposes.

TRIGGER Turn-based vs. concurrent behavior: In turn-based sessions, disconnection of the current-turn player pauses the game. In concurrent sessions, remaining players can continue — disconnected player's state is preserved for when they return.

TRIGGER (Reconnect) Student navigates back / network restores: Browser re-establishes WebSocket. Server validates session is still active. PARTICIPANT_RECONNECTED event fires. Current state is delivered to the reconnected student. Remaining-participant view shows reconnection success.

API Endpoints

API Rejoin after disconnect: POST /api/v1/collaborative-sessions/{session_id}/join — Same endpoint as initial join. Server recognizes existing participant and treats as rejoin. Returns full current session state for catch-up. Per FSM-003 rejoin_participant action.

API WebSocket (session — remaining participants): WebSocket /ws/collab/{session_id} receives participant_disconnected event with { "user_id": "uuid", "display_name": "..." }. Frontend updates roster and shows disconnect notification.

API WebSocket (session — reconnecting student): After POST /join succeeds, student re-subscribes to WebSocket /ws/collab/{session_id}. Server sends the current state snapshot on WebSocket connect: { "type": "state_sync", "state": { ... } }.

API WebSocket (reconnect success broadcast): Server broadcasts participant_reconnected event to all remaining participants: { "user_id": "uuid", "display_name": "..." }. Roster updates and disconnect notification dismisses.

API 5-minute reconnection window (server-side): Server starts a 5-minute timer on disconnect. If the participant does not rejoin within 5 minutes, the server broadcasts a participant_timeout event. If no other participants remain, ALL_DISCONNECTED fires and the session moves to ending.

State Transitions (FSM-003: Collaborative Session Lifecycle)

FSM Disconnect (remaining participants): PARTICIPANT_DISCONNECTED event while in active state. Guard: "At least 1 participant remains." Session stays in active. handle_disconnect action fires: broadcast disconnect event, start 5-min reconnection window.

FSM Reconnect: PARTICIPANT_RECONNECTED event while in active. rejoin_participant action: send current state to reconnected participant, broadcast rejoin event. Session remains in active.

FSM All disconnected: PARTICIPANT_DISCONNECTED event where no participants remain → guard fails → ALL_DISCONNECTED event fires → session moves to ending.

FSM 30-minute inactivity timeout (edge case): If a disconnected player is the only one and the reconnection window expires (they are also the last player), the 30-min inactivity timeout fires → ending.

User Exit Paths

  • Automatic reconnection succeeds: Reconnecting overlay dismisses. Student is back in the session. Roster shows green dot. Other participants see "[Student] reconnected."
  • Reconnection fails (5 min timeout): Disconnected student sees "You couldn't reconnect in time." Button: "Return to chat" → returns to normal student-chat view. Their conversation history includes a note: "You were disconnected from the collaborative session."
  • Remaining participant waits: No action required. Session continues if concurrent. If turn-based, game is paused. Timer counts down. If 5 min passes with no reconnect, session continues with reduced participants or ends if only 1 remains and the session requires 2.
  • Session ends naturally during disconnect wait: If teacher ends session or activity completes while participant is disconnected, the disconnected student receives the session summary when they next connect.

Error Recovery

ERROR POST /join fails on reconnect (5xx): Reconnecting overlay shows: "Having trouble reconnecting. Trying again..." Retries automatically up to 3 times with exponential backoff. If all retries fail: "Could not reconnect. Return to chat." Error logged.

ERROR State sync fails after rejoin: If the WebSocket state_sync message is missing or malformed, the reconnected student sees the app in a loading state. After 5 seconds timeout, an error card shows: "Something went wrong catching you up. Try reloading." Reload triggers a fresh POST /join.

ERROR Multiple simultaneous disconnects: FSM handles each independently. Guard condition "at least 1 participant remains" is checked atomically. If multiple disconnects arrive in the same second, the last check determines if the session moves to ALL_DISCONNECTED.

ERROR Reconnect succeeds but session already ended: POST /join returns 404 (session not found) or 410 (session ended). Show: "The collaborative session has ended. Here's what happened while you were away:" followed by the session summary card.

Accessibility Notes

  • Disconnect notification uses role="status" and aria-live="polite": "[Student Name] disconnected. Waiting for them to reconnect." Not assertive — it's informational, not urgent for the remaining student.
  • Countdown timer uses aria-live="polite" and announces at 4 minutes, 2 minutes, and 1 minute remaining (not every second). Reads: "3 minutes left for [Student Name] to reconnect."
  • Participant roster uses role="list". Each participant: role="listitem" with aria-label="[Name] — connected" or aria-label="[Name] — reconnecting".
  • Reconnecting overlay on disconnected student's view uses role="dialog" (it blocks the session). Spinner has aria-label="Reconnecting to session" and role="progressbar" with aria-valuetext="Attempting to reconnect...".
  • "Return to chat" button in timeout state has aria-label="Return to your individual chat — collaborative session ended".
