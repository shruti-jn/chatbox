Data Retention & Deletion

Screen: data-deletion | FR: FR-SAFE-005 | Persona: PER-003 (Dr. Patel the District Administrator) + Parent | Journeys: JRN-005

Layout Zones

1
Retention Policy Configuration (District Admin view) — Current policy display: "Data is retained for [12] months after the end of the school year." Edit button opens a form with a retention period selector (6, 12, 24 months — configurable within district's DPA terms). Save button. Note: "Changes apply to new data only. Existing data follows the policy at time of collection." Last updated timestamp and updated-by user.
2
Deletion Request Log (District Admin view) — Table of outstanding and completed deletion requests: Student name (anonymized after deletion — shows "Student [ID]"), request date, status (Pending / In Progress / Completed), completion date, confirmation number. Filter by status. Export as CSV for audit.
3
Parent Data Deletion Request Form (Parent/Guardian view — separate URL) — Accessed via a link in the parent consent email or a public-facing URL. Fields: Student name, student's school, parent/guardian email, reason (optional). Verification step: parent must verify identity (same method used for COPPA consent — knowledge-based auth or text-plus). Submit button. SLA disclosure: "Your request will be processed within 30 days."
4
Automated Deletion Audit Log (District Admin, read-only) — Log of automated deletions triggered by retention policy expiry. Each row: batch deletion date, records deleted count, deletion audit log ID. Audit logs are retained independently of student data (as required by FERPA).

Trigger Conditions

TRIGGER (Admin view) District admin navigates to compliance settings: Available under Admin Dashboard → Compliance → Data Retention. Requires district_admin role. Protected route.

TRIGGER (Parent deletion form) Parent submits deletion request: Parents access this via a link in consent notification emails, or by navigating to /privacy/delete-data (public, no auth required before form submission — auth via verification step within form).

TRIGGER (Automated deletion) Retention policy expiry (server-side): A scheduled job runs nightly. Checks all conversation records against the district's configured retention period. Records past their retention date are queued for permanent deletion. No user action triggers this — it is fully automated.

API Endpoints

API Get retention policy: GET /api/v1/admin/district/{district_id}/retention-policy — Returns { "retention_months": 12, "updated_at": "...", "updated_by": "user_id" }. Called on page load for admin view.

API Update retention policy: PUT /api/v1/admin/district/{district_id}/retention-policy — Body: { "retention_months": 12 }. Per AC-SAFE-005-01. Response: updated policy object. Logs audit event.

API Parent deletion request: POST /api/v1/consent/delete-data — Body: { "student_name": "...", "school_name": "...", "parent_email": "...", "verification_token": "..." }. Returns { "request_id": "uuid", "estimated_completion": "ISO8601" }. Confirmation email sent to parent. Per AC-SAFE-005-02: all conversation data, app state, and analytics events deleted within 30 days.

API Get deletion requests (admin): GET /api/v1/admin/district/{district_id}/deletion-requests — Returns paginated list of deletion requests with status. Used for the deletion request log table.

API Get audit log: GET /api/v1/admin/district/{district_id}/deletion-audit-log — Returns automated deletion batch records. Exported as CSV for district's own records.

State Transitions

FSM Deletion request lifecycle: pending → processing (job picks up request, within 30 days) → completed. A confirmation email is sent to the parent when completed. Audit log entry is created regardless of success or failure.

FSM COPPA Consent FSM-004 (related): If a parent requests data deletion, the student's COPPA consent record (FSM-004) transitions to revoked state. The initiate_data_review action is triggered, flagging the student data for retention policy review.

FSM Retention policy change: No FSM state change. Retention policy is a configuration record, not a FSM. The automated deletion job reads this config on each nightly run.

User Exit Paths

  • Save retention policy: PUT fires → success toast "Retention policy updated" → stays on page. Log entry in audit log table updates.
  • Submit deletion request (parent): POST fires → confirmation page: "Your request (ID: XXXX) has been received. We will delete [Student Name]'s data within 30 days and send you a confirmation." No navigation back to form.
  • Cancel retention policy edit: Form collapses. No API call. Returns to read-only display of current policy.
  • Export audit log (CSV): Browser initiates file download. User remains on page.
  • Back / breadcrumb (admin): Returns to Admin Dashboard compliance section.

Error Recovery

ERROR PUT retention policy fails: Inline error: "Could not update retention policy. Try again." Current policy is unchanged. Admin can retry.

ERROR Deletion request POST fails: Inline error on parent form: "Request could not be submitted. Please try again or contact your district at [admin email]." Form data is preserved. Request ID is NOT generated — parent must resubmit.

ERROR Identity verification fails (parent): Show "We could not verify your identity. Please try again. If this persists, contact [support email]." Max 3 attempts before a 24-hour lockout on the form (anti-abuse). After lockout: "Too many failed attempts. Please contact your district administrator."

ERROR Automated deletion job failure: Logged as a critical system event. Retried automatically for 3 days. After 3 days, a compliance alert is sent to the platform admin. The deletion request status shows "Delayed" to the district admin. Parent receives an update email.

Accessibility Notes

  • All form fields have associated <label> elements. No placeholder-only labels.
  • Deletion request table has a <caption> describing its content. Sortable columns use aria-sort.
  • The retention period selector uses a <select> with a visible label and aria-describedby pointing to the explanatory note about what changes apply to.
  • Verification step (parent form): each step in the multi-step flow updates a progress indicator with aria-label="Step 2 of 3: Verify your identity".
  • Success confirmation page uses role="status" to announce the confirmation to screen readers.
