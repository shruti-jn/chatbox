App Resource Limit Warning

Screen: app-resource-limit | FR: FR-APP-005 | Personas: PER-001 (Student), PER-004 (Developer) | Journeys: JRN-001, JRN-006

Layout Zones

1
Resource Error Card (in chat thread — Student view) — Replaces the active app card. Amber/red error styling. Icon: gauge/warning. Message: "This app isn't responding properly. You can try reloading it or continue chatting." Two buttons: "Reload App" and "Dismiss". App name shown. No technical details exposed to student (memory MB counts, request rates are developer-facing only).
2
Resource Limits Threshold States (in-card, progressive) — Three escalating states shown within the app card during normal operation:
  1. Warning (80% of limit): Subtle amber indicator in app card corner. No user action needed.
  2. Critical (95% of limit): App card border turns amber. Small badge: "App running slow." Save progress button appears if app supports state save.
  3. Terminated (limit exceeded): Full error card shown (zone 1).
3
Developer Debug View (developer sandbox only) — In sandbox environments, the error card expands to show: resource type exceeded (memory / network requests), current usage vs. limit (e.g., "52MB / 50MB memory"), timestamp, instance_id. A "View resource logs" button links to the developer console. This is never shown in student-facing production environments.

Trigger Conditions

TRIGGER (Warning — 80%) Server-side resource monitor detects 80% threshold: Memory usage > 40MB or request rate > 80/min. Platform sends app_event via WebSocket with type resource_warning. Client renders amber indicator on app card.

TRIGGER (Critical — 95%) Memory > 47.5MB or request rate > 95/min: Platform sends app_event with type resource_critical. App card border changes. "App running slow" badge appears.

TRIGGER (Terminated) Limit exceeded: Memory > 50MB or request rate > 100/min. Per AC-APP-005-01. RESOURCE_EXCEEDED event fires on FSM-002. Platform terminates the iframe immediately. App card transitions to error state. terminate_for_resources action: kills iframe, shows resource error message, logs event. Per FR-APP-005.

TRIGGER (Size constraint) Iframe dimensions: If an app attempts to resize its iframe beyond 600px width (desktop) or 400px (mobile), the platform clips it and sends a resource_violation log event. Not a termination — just enforcement. Developer sees the violation in sandbox logs.

API Endpoints

API WebSocket resource event (student): WebSocket /ws/chat/{session_id} receives app_event with types: resource_warning, resource_critical, resource_terminated. Each includes instance_id and (for developer sandbox) resource_type and current_usage.

API Reload app (student action): POST /api/v1/apps/{app_id}/tools/{tool_name}/invoke with conversation_id — creates a new app instance. This is the same endpoint as initial invocation. A fresh instance starts with a clean state (terminated instance's state is logged but not restored).

API Get app events (developer): GET /api/v1/apps/{app_id}/events — Returns resource violations, health check failures, security incidents. Used in developer sandbox to debug. Per the API contract operationId: getAppEvents.

API Resource log (server-side): Termination events are automatically logged internally. No client-initiated call. The log includes: instance_id, app_id, resource_type, usage_at_termination, conversation_id (for audit).

State Transitions (FSM-002: App Instance Lifecycle)

FSM Normal operation → resource exceeded: From active state: RESOURCE_EXCEEDED event → terminated. terminate_for_resources action fires: kills iframe, shows resource error card, logs event. This is a terminal transition — the instance cannot be resumed.

FSM Student clicks "Reload App": This creates a NEW instance — FSM-002 starts from registered → TOOL_INVOKED → loading → active. The old terminated instance remains in terminated state in the log.

FSM Student clicks "Dismiss": DISMISS event on the error card instance → terminated (already there, no transition needed). The error card is removed from the conversation view. Conversation returns to plain chat.

FSM Student conversation FSM-001: APP_ERROR event in FSM-001 (triggered by resource termination) → if in app_embedded state → transition to active. handle_app_error action fires: AI acknowledges the failure in the conversation.

User Exit Paths

  • "Reload App": New app instance created via POST /invoke. Loading skeleton appears. App loads fresh state. Old error card is replaced by new loading card. Student loses in-progress state from the terminated session.
  • "Dismiss": Error card is removed from view. Chat input is the active surface. The AI sends a message acknowledging the issue. Student continues chatting without the app.
  • Ignore card and continue chatting: Student can type in the chat input at any time. The error card persists in the thread as a record. The AI is aware the app terminated and will not reference it as if it were still active.
  • Session end: SESSION_END event. Error card persists in conversation history (it's a message in the thread, not a transient overlay).

Error Recovery

ERROR Reload fails (POST /invoke returns 4xx): Show inline error on the new loading card: "Could not reload [App Name]. Try again later." Error card replaces loading skeleton. Chat continues.

ERROR Reload fails (new instance also hits resource limit immediately): Same resource termination flow as before. After 2 consecutive immediate terminations, the error card shows: "This app keeps running into problems. It may need to be fixed by its developer. You can continue chatting." No more reload button shown.

ERROR WebSocket unavailable during termination: The iframe is killed server-side. When student's WebSocket reconnects, the client queries session state and renders the error card retroactively. Student may have had a period where the app appeared frozen before the error card appeared.

ERROR Developer sandbox resource log unavailable: Developer sees "Resource logs temporarily unavailable" on the debug panel. The resource termination still occurred — the UI is a display failure only.

Accessibility Notes

  • Resource error card uses role="alert" and is announced when it appears: "[App Name] stopped working. You can reload it or continue chatting."
  • Warning and critical indicators (amber dot, border color) use both color and a text label: "App running slow" is visible text, not just a color change. WCAG 1.4.1 compliance.
  • "Reload App" button has aria-label="Reload [App Name]". "Dismiss" has aria-label="Dismiss error and continue chatting".
  • After reload, focus moves to the new loading skeleton's region. Screen reader announces: "[App Name] is loading."
  • Developer debug panel (sandbox) collapses by default. Expandable via keyboard with a "Show resource details" toggle button.
