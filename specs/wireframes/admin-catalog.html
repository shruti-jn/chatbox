Admin App Catalog

Screen: admin-catalog | Persona: PER-003 (Dr. Patel the District Admin) | Journey: JRN-005

Information Hierarchy

1
Summary Stats Row (top) — Four metric cards showing approved/pending/suspended counts and total sessions. The pending count is the primary action driver — if pending > 0, the admin has work to do.
2
App Table (center) — The main working surface. All apps listed with status, auth type, developer, review date, session count, and action buttons. Sorted by status (pending first) by default.
3
Filter Bar (above table) — Search, status filter, auth type filter. Enables quick narrowing when catalog grows beyond 20 apps.

Focal Point

FOCAL The "Pending Review" stat card draws initial attention due to its amber coloring. If pending > 0, the admin's primary task is to review those apps. The table rows with pending status are the secondary focal point.

Above-the-fold: Stats row + filter bar + first 4-5 table rows visible without scrolling.

Banned Dominance Patterns

  • No large "Add New App" hero CTA — app submission comes from developers, not admins
  • No card-based app gallery layout — table is more scannable for admin review workflows
  • Stats cards must not be so large they push the table below the fold

Interactive States

CLICK Review button: Opens a detail modal/page showing the app's manifest: declared permissions, data access requirements, content types, developer info, privacy policy link, and security audit results. Admin can approve/reject from this view.

CLICK Approve button: Confirms via dialog ("Approve Spotify Playlist for all district teachers?"). On confirm, app status changes to Approved (green badge). Row moves in sort order.

CLICK Reject button: Opens dialog with required rejection reason (text field). Rejection reason is sent to developer.

CLICK Suspend button: Confirms via dialog ("Suspend Drawing Canvas? Active sessions will end."). Changes status to Suspended. Active sessions are terminated gracefully.

CLICK Reinstate button: Confirms and changes status back to Approved.

FILTER Search input: Filters table rows client-side by app name or developer name. Debounced 300ms.

FILTER Status dropdown: Filters table rows by status. Immediate, no debounce.

API Integration Annotations

API App list: GET /api/v1/admin/apps?district_id={id} — returns all apps with status, auth type, developer, review dates, session counts. Supports query params for status filter.

API Summary stats: GET /api/v1/admin/apps/stats?district_id={id} — returns aggregate counts per status and total weekly sessions.

API Approve app: PATCH /api/v1/admin/apps/{app_id}/approve — changes status to approved.

API Reject app: PATCH /api/v1/admin/apps/{app_id}/reject — body includes { "reason": "..." }.

API Suspend app: PATCH /api/v1/admin/apps/{app_id}/suspend — terminates active sessions and changes status.

API Reinstate app: PATCH /api/v1/admin/apps/{app_id}/reinstate — changes status back to approved.

API App detail (review): GET /api/v1/admin/apps/{app_id}/manifest — full app manifest with permissions, data declarations, security audit.

Responsive Behavior

MOBILE 375px Table transforms to card list (one card per app). Stats row wraps to 2x2 grid. Filter inputs stack vertically. Admin workflow is primarily desktop — mobile is for quick status checks only.

DESKTOP 1280px Full table layout with all columns visible. Stats row horizontal. This is the primary viewport for admin users.

Persona Fit

PERSONA District administrators manage app catalogs on quarterly/annual review cycles, not daily. The table format matches their mental model: scan, filter, take action. The stats row provides instant situational awareness ("3 pending" means "I have work to do"). Action buttons are per-row because admin decisions are per-app, not batch. The review workflow (Review -> detail -> Approve/Reject) mirrors procurement vetting processes admins already follow.
